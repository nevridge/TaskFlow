# Technical Brief: Associate a Journal Log Entry with a Task

**Date:** 2026-05-31  **Status:** Approved

## 1. Overview
Lets a user optionally link a `JournalLogEntry` to any existing `TaskItem`, indicating that the log entry represents work done on that task. The association is a nullable FK on `JournalLogEntry`, settable at creation time or via update, clearable by passing `null`. When the linked task is deleted, `OnDelete(SetNull)` nulls the FK — but a denormalized `LinkedTaskTitleSnapshot` field preserves the last-known title so a "deleted" badge can still be shown.

## 2. Data Model Changes
Modify `TaskFlow.Api/Models/JournalLogEntry.cs`:
- Add `public int? TaskItemId { get; set; }`
- Add `public TaskItem? TaskItem { get; set; }`
- Add `public string? LinkedTaskTitleSnapshot { get; set; }` — populated at link time; enables "deleted" detection when `TaskItemId == null && LinkedTaskTitleSnapshot != null`

`TaskFlow.Api/Data/TaskDbContext.cs` — add in a new `modelBuilder.Entity<JournalLogEntry>()` block:
```csharp
entity.HasOne(l => l.TaskItem)
      .WithMany()
      .HasForeignKey(l => l.TaskItemId)
      .OnDelete(DeleteBehavior.SetNull);
entity.HasIndex(l => l.TaskItemId);
```

Migration: `dotnet ef migrations add AddJournalLogTaskLink --project TaskFlow.Api`
All columns are nullable — additive, no backfill, safe on existing data.

## 3. API Contract

**`POST /api/v1/JournalEntries/{entryId}/logs`**
Request `CreateJournalLogEntryDto`: `{ content: string (required), taskItemId?: int | null }`
Response `201` `JournalLogEntryResponseDto` (below)
Status codes: `201`, `400` (invalid content or non-existent taskItemId), `404` (journal entry not found)

**`PUT /api/v1/JournalEntries/{entryId}/logs/{id}`**
Request `UpdateJournalLogEntryDto`: `{ content: string (required), taskItemId?: int | null }`
Response `200` with the same DTO
Status codes: `200`, `400`, `404`

**`JournalLogEntryResponseDto`** gains three new fields:
- `taskItemId?: number | null`
- `linkedTaskTitle?: string | null` — live title when task exists; falls back to `LinkedTaskTitleSnapshot` when deleted
- `linkedTaskDeleted: boolean` — `true` when `TaskItemId == null && LinkedTaskTitleSnapshot != null`

GET endpoints are unchanged in route; they now return the new fields.

## 4. Backend Changes

**`JournalLogEntriesController`:** inject `ITaskRepository taskRepo`. In `Create` and `Update`: if `dto.TaskItemId` has a value, call `await taskRepo.GetByIdAsync(value)` — return `BadRequest` if null; else set `log.TaskItemId` and `log.LinkedTaskTitleSnapshot = task.Title`. On `Update` with `null` taskItemId, clear both fields. Update `ToDto` to map all three new fields (compute `linkedTaskDeleted` and title fallback).

**`JournalLogEntryRepository`:** add `.Include(l => l.TaskItem)` to read queries so the live title is available for the DTO.

**`JournalLogEntryValidator`:** add shape-only rule: `RuleFor(e => e.TaskItemId).GreaterThan(0).When(e => e.TaskItemId.HasValue)`. Existence check stays in the controller.

**No new extension methods** — `ITaskRepository` is already registered.

## 8. Out of Scope
- Filtering the typeahead to open tasks only (all task statuses are shown)
- Events/audit trail on log entry deletion
- Restricting linkable tasks to the day's journal todos (backend permits any task)
- Soft-delete for tasks
